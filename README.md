## Overview

**ePay** is a Bangladeshi mobile financial services (MFS) application similar to bKash or Nagad. It allows users to manage their digital wallet вҖ” send money, cash out via agents or ATMs, add money from banks or cards, pay utility bills, and access remittance services.

## Architecture

This project follows a **layered clean architecture** pattern:

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ           UI Layer              в”Ӯ  Screens + Widgets
в”Ӯ   (modules/*/screens/)          в”Ӯ  Pure Flutter widgets, no business logic
в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө
в”Ӯ        Controller Layer         в”Ӯ  GetxController per module
в”Ӯ   (modules/*/controllers/)      в”Ӯ  Handles state, user events, navigation
в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө
в”Ӯ        Repository Layer         в”Ӯ  Mediates between controller and provider
в”Ӯ   (data/repositories/)          в”Ӯ  Single source of truth for data access
в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө
в”Ӯ         Provider Layer          в”Ӯ  Raw API calls via NetworkService
в”Ӯ   (data/providers/)             в”Ӯ  All mock endpoints вҖ” easily replaceable
в”ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Ө
в”Ӯ          Core Layer             в”Ӯ  Constants, theme, services, utils
в”Ӯ   (core/)                       в”Ӯ  No dependencies on other layers
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

### Key Principles Applied

- **Separation of Concerns** вҖ” each layer has a single responsibility
- **Dependency Injection** вҖ” via GetX bindings; controllers receive repositories through constructors
- **Repository Pattern** вҖ” controllers never call providers directly
- **Mock-First API Design** вҖ” all providers return mock data; real endpoints are drop-in replacements
- **Single Responsibility** вҖ” each file does exactly one thing

## Features

### вң… Authentication
- Sign Up Welcome screen with animated illustration and floating colored dots
- Sign Up with phone number + 4-digit PIN + confirm PIN validation
- OTP verification with 4-box input, auto-focus between boxes, 30-second resend countdown
- Login with phone number + 6-digit PIN + biometric fingerprint placeholder
- Language toggle (English / аҰ¬аҰҫаҰӮаҰІаҰҫ) on every auth screen

### вң… Home Dashboard
- Personalized greeting with user avatar and name
- Real-time balance display with show/hide toggle
- Points badge with trophy icon (e.g. 1,972 Points)
- 8-item services grid with "See More / See Less" toggle
- Pay Bill section вҖ” 8 utility categories (Electricity, Gas, Water, Internet, Telephone, Credit Card, Govt. Fees, Cable Network)
- Remittance partners horizontal scroll (Payoneer, PayPal, Wind, Wise)

### вң… Cash Out
- **Agent Tab** вҖ” input agent number manually, QR code scan button, recent contacts, all contacts
- **ATM Tab** вҖ” available balance display, live bank search, partner bank cards with branch info
- Confirm screen вҖ” agent number display, large centered amount, available balance, disabled/enabled confirm button based on input
- Success dialog вҖ” orange circular checkmark with withdrawal amount highlighted in gold

### вң… Send Money
- Search by name or number input field with contact book icon
- Recent contacts and All contacts sections
- Confirm screen вҖ” contact number, large centered amount, real-time remaining balance
- Success dialog вҖ” two-hands flying coins custom `CustomPainter` illustration with amount highlighted

### вң… Add Money
- **Bank to Ekpay Tab** вҖ” Bank Account + Internet Banking radio options

## State Management

GetX is used throughout with the following conventions:

### Observable Pattern
```dart
// in controller
Rx<RxStatus> status = Rx<RxStatus>(RxStatus.loading());
RxList<ContactModel> contacts = <ContactModel>[].obs;
final RxBool isLoading = false.obs;
final RxString selectedPhone = ''.obs;

// in screen вҖ” Obx reads .value at the exact point of use
Obx(() => Text(controller.selectedPhone.value))
```

### Obx Placement Rule
Every `.value` read **must be directly inside** its own `Obx` closure. Observables are read at the lowest possible widget level to minimise unnecessary rebuilds.

```dart
// вң… correct вҖ” .value read directly inside Obx scope
Obx(() => Text(controller.name.value))

// вқҢ wrong вҖ” .value hidden inside child widget, outside Obx tracking scope
Obx(() => _ChildWidget(controller: controller))
// causes "improper use of GetX" warning if _ChildWidget reads .value internally
```

### Controller Lifecycle
- Controllers registered via `Bindings` вҖ” one binding class per module
- `Get.lazyPut` so controllers are created only when their screen is first visited
- Controllers automatically disposed when their screen is removed from the stack
- `onClose()` overridden to dispose `TextEditingController`, `FocusNode`, and `Timer`

## Networking

### NetworkResult\<T\>
All API calls return a typed `NetworkResult<T>` wrapper:

```dart
NetworkResult.success(data, statusCode, message)  // isSuccess = true
NetworkResult.failure(statusCode, message)         // isSuccess = false
```

### NetworkService
A singleton Dio client with centralized request/response logging:

```dart
final result = await NetworkService.instance.get(endpoint);
final result = await NetworkService.instance.post(endpoint, body);
```

### CacheService
A `GetStorage` wrapper for lightweight local persistence:

```dart
CacheService.instance.write(key, value);
CacheService.instance.read(key);
CacheService.instance.remove(key);
```

### Mock API Pattern
All providers use structured mock data that mirrors the expected real API response shape. Replacing them requires changes in **one file only** вҖ” the provider.

```dart
// current mock
Future<NetworkResult<dynamic>> login({required String phone, required String pin}) async {
  return NetworkResult.success({'id': '1', 'name': 'RAHUL', ...}, 200, 'Success');
}

// real endpoint вҖ” zero changes to controller or repository
Future<NetworkResult<dynamic>> login({required String phone, required String pin}) async {
  return NetworkService.instance.post('$_base/auth/login', {'phone': phone, 'pin': pin});
}
```

## Replacing Mock APIs

**Step 1** вҖ” Set the real base URL in the provider file:
```dart
static const String _base = 'https://your-real-api.com/api/v1';
```

**Step 2** вҖ” Replace the mock return with a real Dio call:
```dart
// before (mock)
Future<NetworkResult<dynamic>> login({...}) async {
return NetworkResult.success({'name': 'RAHUL', ...}, 200, 'Success');
}

// after (real)
Future<NetworkResult<dynamic>> login({...}) async {
return NetworkService.instance.post('$_base/auth/login', {
'phone': phone, 'pin': pin,
});
}
```

**Step 3** вҖ” Update the model `fromJson` if the real API response shape differs.

вң… Controllers, repositories, and screens require **zero changes**.

## Code Conventions

| Convention | Example |
|-----------|---------|
| All inline comments in lowercase | `// balance visibility toggle` |
| Private widgets prefixed with `_` | `class _BalanceCard` |
| All constants in dedicated classes | `AppColors.primary`, `AppStrings.login` |
| One screen per file | `home_screen.dart` |
| Screen-specific widgets in `screens/widgets/` | organized per module |
| Cross-screen widgets in `common_widgets/` | `AppButton`, `LoadingStateWidget` |
| Controllers receive repositories via constructor | `HomeController(this._repository)` |
| Observables named descriptively | `isBalanceVisible`, `selectedPhone` |
| Each `Obx` reads only the `.value` it directly needs | no child widget hides `.value` from parent `Obx` |
| No hardcoded asset path strings | always use generated `Assets.xxx` class |
| No hardcoded color, size, or text literals | always use `AppColors`, `AppSpacing`, `AppTypography`, `AppStrings` |
